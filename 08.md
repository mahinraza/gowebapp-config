## The Big Picture

```
CLUSTER
│
├── 🏗️  Infrastructure Chart (installed ONCE — cluster-wide)
│   ├── cert-manager
│   ├── ClusterIssuer (letsencrypt-prod)
│   ├── ClusterIssuer (letsencrypt-staging)
│   ├── ingress-nginx
│   ├── external-dns
│   └── monitoring stack
│
├── 📦  App Team A
│   ├── my-app-1  (just references letsencrypt-prod)
│   └── my-app-2  (just references letsencrypt-prod)
│
├── 📦  App Team B
│   ├── service-x  (just references letsencrypt-prod)
│   └── service-y  (just references letsencrypt-prod)
│
└── 📦  App Team C
    └── api-gateway  (just references letsencrypt-prod)
```

**ClusterIssuer is a cluster-scoped resource** — one issuer serves all namespaces and all applications simultaneously.

## Responsibility Split

```
┌─────────────────────────────────────────────────────────────────┐
│                    PLATFORM / OPS TEAM                          │
├─────────────────────────────────────────────────────────────────┤
│  Owns infrastructure chart                                      │
│  ✅ Installs cert-manager                                       │
│  ✅ Creates ClusterIssuer (letsencrypt-prod / staging)          │
│  ✅ Manages ingress-nginx, external-dns                         │
│  ✅ Sets cluster-wide policies                                  │
│  ✅ Rotates Route53 / Cloudflare credentials for dns01          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    APPLICATION TEAMS                            │
├─────────────────────────────────────────────────────────────────┤
│  Own their app charts                                           │
│  ✅ Reference ClusterIssuer by name in ingress annotation       │
│  ✅ Specify their domain/hostname                               │
│  ✅ Pick staging vs prod issuer per environment                 │
│  ❌ Never create ClusterIssuers                                 │
│  ❌ Never touch cert-manager config                             │
└─────────────────────────────────────────────────────────────────┘
```

## What App Teams Actually Write

Every app team just does this in their ingress — **that's it**:

```yaml
# apps/team-a/my-app-1/templates/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app-1
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"   # ← just a name, nothing else
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - my-app-1.example.com
      secretName: my-app-1-tls          # cert-manager auto-creates this secret
  rules:
    - host: my-app-1.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-app-1
                port:
                  number: 80
```

App teams don't know or care **how** certs are issued — they just reference the issuer name. The platform team handles everything underneath.

## Multi-Environment Setup (Staging vs Prod Clusters)

In most companies there are separate clusters per environment:

```
staging-cluster/
  └── infrastructure chart → ClusterIssuer: letsencrypt-staging
                          → ingress-nginx
                          → external-dns (staging zone)

prod-cluster/
  └── infrastructure chart → ClusterIssuer: letsencrypt-prod
                          → ingress-nginx
                          → external-dns (prod zone)
```

App teams use `values-staging.yaml` and `values-prod.yaml`:

```yaml
# values-staging.yaml
ingress:
  host: my-app.staging.example.com
  tls:
    clusterIssuer: letsencrypt-staging   # ← staging

# values-prod.yaml
ingress:
  host: my-app.example.com
  tls:
    clusterIssuer: letsencrypt-prod      # ← prod
```

## TL;DR

| Question | Answer |
|---|---|
| Who installs infrastructure chart? | Platform / Ops team |
| How many times is it installed? | Once per cluster |
| How many apps does it serve? | All apps in the cluster |
| What do app teams do? | Just reference `letsencrypt-prod` by name |
| Per-environment? | Separate infra chart per cluster (staging/prod) |

Think of the **infrastructure chart like shared utilities in a building** — one electricity connection, one water supply, everyone uses it. App teams don't wire their own electricity, they just plug in.
